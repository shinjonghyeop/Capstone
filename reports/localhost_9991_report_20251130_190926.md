# 보안 진단 보고서

## 1. 경영진 요약 (Executive Summary)

**목적**: 비기술직 경영진을 위한 핵심 요약

대상 시스템(localhost:9991)에 대한 웹 취약점 스캔 결과, **심각 (Severe)** 수준의 보안 위험이 확인되었습니다. 총 54개의 취약점 중 고위험(High) 13개, 중간(Medium) 14개가 발견되었으며, 특히 로컬 파일 포함(Local File Inclusion, LFI) 및 경로 탐색(Path Traversal) 취약점이 다수 집중되어 있습니다. 계산된 총 위험 점수는 120/100으로, 이는 즉각적인 개입과 광범위한 보안 강화가 필요함을 강력히 시사합니다.

**발견 취약점 개수**: 54개 (Critical: 0, High: 13, Medium: 14, Low: 27, Info: 0)

**비즈니스 영향 분석**:
*   **데이터 유출 가능성**: LFI 및 경로 탐색 취약점은 공격자가 서버 내 민감한 설정 파일, 사용자 데이터, 소스 코드 등을 쉽게 열람할 수 있게 하여 심각한 데이터 유출로 이어질 수 있습니다. 이는 개인정보보호법(GDPR, CCPA 등) 위반 및 기업 평판에 치명적인 손상을 줄 수 있습니다.
*   **서비스 중단 위험**: 발견된 취약점들은 원격 코드 실행(Remote Code Execution, RCE)으로 발전할 가능성이 높아, 공격자가 시스템을 완전히 장악하여 서비스 중단, 웹사이트 변조, 악성코드 배포 등을 야기할 수 있습니다.
*   **컴플라이언스 위반 가능성**: 암호화되지 않은 채널을 통한 민감 정보 전송, 평문 비밀번호 사용, 부적절한 보안 헤더 설정 등은 주요 보안 표준 및 규제(예: OWASP Top 10, ISO 27001) 준수 미달로 이어져 법적 및 재정적 불이익을 초래할 수 있습니다.

**우선순위 조치사항**:
1.  **🔴 긴급: 로컬 파일 포함(LFI) 및 경로 탐색(Path Traversal) 취약점 즉시 제거**: 서버 내 민감 정보 유출 및 시스템 장악을 방지하기 위해 사용자 입력 값에 대한 엄격한 검증 및 파일 경로 조작 방어 로직을 구현해야 합니다.
2.  **🟡 중요: 모든 웹 통신에 HTTPS 적용 및 평문 비밀번호 전송 문제 해결**: 사용자의 모든 데이터(특히 로그인 정보)가 도청 및 탈취될 위험을 제거하기 위해 웹 서버에 SSL/TLS 인증서를 적용하고, HTTP를 통한 접속은 HTTPS로 강제 전환해야 합니다.
3.  **🟡 중요: 크로스 사이트 스크립팅(XSS) 취약점 수정**: 사용자 세션 탈취, 악성 코드 주입 등 클라이언트 측 공격을 방지하기 위해 사용자 입력 값에 대한 출력 인코딩을 철저히 적용해야 합니다.
4.  **🟢 일반: 보안 헤더(CSP, X-Frame-Options, X-Content-Type-Options) 설정**: 클릭재킹, MIME 타입 스니핑 등 다양한 클라이언트 측 공격을 완화하고, 전반적인 웹 보안 수준을 향상시키기 위해 적절한 보안 헤더를 구성해야 합니다.

**예상 위험 시나리오**:
공격자는 `lvl1.php` 엔드포인트의 LFI 취약점을 악용하여 서버의 `passwd` 파일이나 데이터베이스 설정 파일 등 핵심 정보를 빼냅니다. 이 정보를 통해 시스템에 접근 가능한 계정 정보를 획득하거나, 추가적인 취약점(예: 원격 코드 실행)과 연계하여 서버를 완전히 장악합니다. 서버가 장악되면 웹사이트가 변조되거나, 악성 코드가 배포되어 방문자들에게 피해를 줄 수 있습니다. 또한, XSS 취약점은 사용자들에게 피싱 페이지를 보여주거나 세션 정보를 탈취하여 사용자 계정을 도용할 수 있습니다. 모든 통신이 암호화되지 않아 사용자 로그인 정보가 네트워크를 통해 노출될 위험도 상존합니다.

## 2. 기술적 상세 분석 (Technical Deep Dive)

본 섹션에서는 스캔 결과 발견된 취약점들을 기술적인 관점에서 상세히 분석하고, 구체적인 수정 방안을 제시합니다. 특히, 고위험(High) 및 중간(Medium) 취약점 중 영향도가 큰 항목들을 중심으로 다룹니다.

### 2.1 Critical 및 High 위험 취약점

Critical 취약점은 발견되지 않았으나, High 위험 취약점은 13개로 매우 심각합니다. 특히, `/www/FileInclusion/pages/lvl1.php/` 엔드포인트가 다수의 로컬 파일 포함(LFI) 및 경로 탐색(Path Traversal) 취약점의 주요 진입점으로 확인되었습니다. 이는 단일 취약점이 다양한 방식으로 악용될 수 있음을 의미합니다.

#### [High] FHEM 6.0 - Local File Inclusion

*   **취약점 ID**: nuclei-cve-0
*   **카테고리**: CVE
*   **위치**: `http://localhost:9991/www/FileInclusion/pages/lvl1.php/fhem/FileLog_logWrapper`
*   **CVE**: `cve-2020-19360`

**공격 시나리오**:
공격자는 `lvl1.php` 뒤에 오는 경로(`fhem/FileLog_logWrapper`)를 조작하여 `../../../../etc/passwd`와 같은 상대 경로를 삽입합니다. 이로 인해 웹 애플리케이션은 의도치 않게 서버의 `/etc/passwd` 파일을 읽어 응답에 포함시키게 됩니다. 공격자는 이 방법을 통해 시스템의 사용자 계정 정보, 데이터베이스 설정 파일, 애플리케이션 소스 코드 등 민감한 파일을 자유롭게 열람할 수 있습니다.

**기술적 원인**:
`lvl1.php` 또는 그 하위 로직에서 사용자로부터 전달받은 파일 경로를 검증 없이 `include()` 또는 `require()` 함수에 사용하는 것이 주된 원인입니다. `../` (dot-dot-slash) 시퀀스를 제거하거나, 허용된 디렉토리 내의 파일만 포함하도록 제한하는 등의 입력 값 검증(Input Validation)이 부재합니다. 또한, 취약한 FHEM 컴포넌트가 웹 서버에 노출되어 있을 가능성이 있습니다.

**영향 범위**:
서버 파일 시스템 전체의 읽기 권한이 탈취될 수 있습니다. 이는 데이터 유출(Data Leakage), 설정 파일 노출, 잠재적인 원격 코드 실행(Remote Code Execution, RCE)으로 이어져 시스템 전체가 장악될 위험이 있습니다.

**수정 방안**:
1.  **입력 값 검증 강화**: 사용자 입력으로 파일 경로를 구성할 경우, `basename()` 함수를 사용하여 디렉토리 정보를 제거하거나, 허용된 파일 목록(whitelist)을 정의하여 해당 목록에 있는 파일만 포함하도록 제한합니다.
    *   **예시 (PHP)**: `if (in_array($file, ['allowed_file1.php', 'allowed_file2.php'])) { include $file; }`
    *   **예시 (PHP)**: `include dirname(__FILE__) . '/' . basename($file);`
2.  **`allow_url_include` 비활성화**: PHP 설정(`php.ini`)에서 `allow_url_include = Off`로 설정하여 원격 파일 포함(RFI) 공격을 방지합니다.
3.  **최소 권한 원칙**: 웹 서버 프로세스가 필요한 최소한의 파일 및 디렉토리에만 접근 권한을 가지도록 설정합니다.
4.  **불필요한 컴포넌트 제거**: 취약한 것으로 알려진 FHEM, GLPI, Joomla 등 불필요한 애플리케이션이나 플러그인은 즉시 제거하거나 업데이트합니다.

#### [High] GLPI plugin Barcode < 2.6.1 - Path Traversal Vulnerability.

*   **취약점 ID**: nuclei-cve-2
*   **카테고리**: CVE
*   **위치**: `http://localhost:9991/www/FileInclusion/pages/lvl1.php/glpi/plugins/barcode/front/send.php`
*   **CVE**: `cve-2021-43778`

**공격 시나리오**:
`lvl1.php`를 통해 `glpi/plugins/barcode/front/send.php`로 요청이 전달될 때, `file` 또는 유사한 파라미터에 `../../../../etc/shadow`와 같은 경로 탐색 페이로드를 삽입하여 서버의 섀도우 파일(사용자 비밀번호 해시 포함)을 탈취합니다.

**기술적 원인**:
GLPI Barcode 플러그인(버전 2.6.1 미만)의 `send.php` 파일에서 파일 경로를 처리하는 과정에서 `../` 시퀀스에 대한 적절한 검증 및 정규화(canonicalization)가 이루어지지 않아 발생합니다. 이 취약점은 `lvl1.php`의 LFI 취약점과 결합되어 더욱 위험해집니다.

**영향 범위**:
LFI와 유사하게 서버의 임의 파일 읽기가 가능하며, 특히 `shadow` 파일과 같은 민감한 인증 정보가 유출될 경우 시스템 계정 탈취로 이어질 수 있습니다.

**수정 방안**:
1.  **GLPI Barcode 플러그인 업데이트/제거**: 즉시 플러그인을 최신 보안 패치 버전(2.6.1 이상)으로 업데이트하거나, 사용하지 않는 경우 제거합니다.
2.  **파일 경로 보안 처리**: `realpath()`와 같은 함수를 사용하여 파일 경로를 정규화하고, 지정된 안전한 디렉토리 외부로 접근하는 시도를 탐지하고 차단합니다.
    *   **예시 (PHP)**: `$baseDir = '/var/www/html/safe_files/'; $requestedFile = realpath($baseDir . $userInputFile); if (strpos($requestedFile, $baseDir) !== 0) { die("Access Denied"); } include $requestedFile;`

#### [High] Centos Web Panel 0.9.8.480 - Local File Inclusion

*   **취약점 ID**: nuclei-cve-12
*   **카테고리**: CVE
*   **위치**: `http://localhost:9991/www/FileInclusion/pages/lvl1.php/admin/index.php`
*   **CVE**: `cve-2018-18323`

**공격 시나리오**:
공격자가 `lvl1.php`를 통해 `admin/index.php`의 취약점을 이용, `file` 파라미터에 LFI 페이로드를 주입하여 CWP 설정 파일이나 로그 파일 등을 읽어낼 수 있습니다. 설명에 따르면 XSS 및 RCE 취약점도 알려져 있어, LFI를 통해 추가적인 공격 발판을 마련할 수 있습니다.

**기술적 원인**:
Centos Web Panel(CWP) 0.9.8.480 버전 자체의 로컬 파일 포함 취약점입니다. CWP가 설치된 환경에서 `lvl1.php`를 통해 CWP의 취약한 엔드포인트에 접근할 수 있게 되면, 해당 취약점들이 악용됩니다.

**영향 범위**:
CWP의 관리자 계정 정보, 시스템 설정, 다른 가상 호스트의 데이터 등 광범위한 정보가 유출될 수 있으며, RCE로 발전 시 웹 서버 및 호스팅된 모든 서비스가 완전히 장악될 수 있습니다.

**수정 방안**:
1.  **CWP 즉시 업데이트 또는 제거**: CWP 0.9.8.480은 오래되고 알려진 취약점이 많은 버전이므로, 최신 보안 패치 버전으로 업데이트하거나, 사용하지 않는 경우 즉시 제거해야 합니다.
2.  **웹 애플리케이션 분리**: `lvl1.php`와 같은 취약한 애플리케이션이 CWP와 같은 중요한 시스템 관리 도구에 접근할 수 없도록 네트워크 및 파일 시스템 레벨에서 격리해야 합니다.

#### [High] Responsive FileManager <9.13.4 - Local File Inclusion

*   **취약점 ID**: nuclei-cve-15
*   **카테고리**: CVE
*   **위치**: `http://localhost:9991/www/FileInclusion/pages/lvl1.php/filemanager/ajax_calls.php`
*   **CVE**: `cve-2018-15535`

**공격 시나리오**:
`lvl1.php`를 통해 `filemanager/ajax_calls.php`로 전달되는 `file` 파라미터에 LFI 페이로드를 주입하여 파일 매니저가 관리하는 파일 외에 서버의 다른 파일을 읽어낼 수 있습니다.

**기술적 원인**:
Responsive FileManager 버전 9.13.4 미만에서 `filemanager/ajax_calls.php`가 외부 입력으로 파일 경로를 구성할 때 적절한 제한 없이 사용하기 때문에 발생합니다. `lvl1.php`의 LFI가 이 취약한 컴포넌트를 호출하는 통로가 됩니다.

**영향 범위**:
파일 매니저가 접근 가능한 모든 파일 및 디렉토리뿐만 아니라, LFI를 통해 서버의 임의 파일에 접근하여 민감 정보가 유출될 수 있습니다.

**수정 방안**:
1.  **Responsive FileManager 업데이트/제거**: 즉시 최신 보안 패치 버전(9.13.4 이상)으로 업데이트하거나, 사용하지 않는 경우 제거합니다.
2.  **파일 접근 권한 제한**: 웹 서버 프로세스가 파일 매니저의 파일 저장 디렉토리 외에는 접근할 수 없도록 권한을 엄격하게 설정합니다.

#### [Medium] Django Debug Page - Cross-Site Scripting (CVE-2017-12794)

*   **취약점 ID**: nuclei-cve-45
*   **카테고리**: CVE / XSS
*   **위치**: `http://localhost:9991/www/XSS/XSS_level1.php/create_user/`
*   **CVE**: `cve-2017-12794`

**공격 시나리오**:
Django 애플리케이션의 `DEBUG=True` 설정이 활성화된 상태에서, 공격자는 특정 오류를 유발하는 URL(`create_user/`와 같은)에 악성 스크립트(`</script><script>alert('XSS')</script>`)를 포함한 요청을 보냅니다. Django의 기술적인 500 디버그 페이지 템플릿의 특정 부분에서 HTML 자동 이스케이프(autoescaping)가 비활성화되어 있어, 이 악성 스크립트가 사용자 브라우저에서 실행됩니다. 이를 통해 공격자는 사용자 세션 탈취, 쿠키 정보 획득, 페이지 변조, 피싱 공격 등을 수행할 수 있습니다.

**기술적 원인**:
Django 1.10.x (1.10.8 미만) 및 1.11.x (1.11.5 미만) 버전의 디버그 페이지 템플릿에서 HTML 자동 이스케이프가 부분적으로 비활성화되어 있어, 신뢰할 수 없는 입력이 그대로 HTML로 렌더링될 수 있습니다. 본 스캔 결과에서는 `DEBUG=True` 상태가 감지되었습니다.

**영향 범위**:
애플리케이션을 사용하는 모든 사용자가 XSS 공격에 노출될 수 있습니다. 사용자 세션 하이재킹(Session Hijacking), 개인 정보 탈취, 웹사이트 변조, 악성 코드 배포 등으로 이어질 수 있습니다.

**수정 방안**:
1.  **운영 환경에서 `DEBUG=False` 설정**: 가장 중요합니다. 운영 환경에서는 절대로 `DEBUG=True`로 설정해서는 안 됩니다. `settings.py`에서 `DEBUG = False`로 변경합니다.
2.  **Django 버전 업데이트**: Django를 보안 패치가 적용된 최신 버전(예: 1.10.8 이상 또는 1.11.5 이상)으로 업데이트합니다.
3.  **Content Security Policy (CSP) 구현**: XSS 공격의 영향을 완화하기 위해 강력한 CSP를 설정하여 외부 스크립트 실행을 제한합니다.
